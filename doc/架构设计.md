把 agent 设计成“确定性工程骨架 + LLM 负责提出优化补丁 + 本地评测器负责裁判”的闭环系统。

**总体架构**
`run.sh` 启动 `agent.py`，agent 最终生成：

- `workspace/engine.py`：真正被评测 import 的 runtime
- `workspace/results.log`：完整记录每轮候选、正确性、吞吐、失败原因，所以每个模块都需要记录日志
- `workspace/output3.md`：总结 agent 的推理、优化路线和最终选择

核心模块建议如下。

**1. 环境与输入探测模块**
负责读取 `/target/model_config.json`、`/target/weights`，本地则回退到 `target/...`。同时生成一份结构化摘要：

- 模型层数、hidden size、heads、kv heads、head dim、vocab size、dtype
- 权重 tensor 名称、shape、dtype、总参数量
- 是否有 CUDA、可用显存、PyTorch/Triton 是否可用
- 如果能看到 trace，就统计 batch size、prompt length、decode steps、remove/insert 模式

这部分不要交给 LLM 猜，agent 自己确定性完成。

**2. Runtime 规格抽取模块**
把作业接口转成硬约束清单，供后续每轮生成和自检使用：

- `create_engine(model_config, weight_dir, device)`
- `prefill` 创建或替换 request state
- `decode` 只能处理已经 prefill 的请求
- `remove` 必须释放状态
- logits 顺序必须严格对应 `request_ids`
- 不能 hard-code hidden 模型参数
- 正确性优先于性能

这份规格会进入 LLM prompt，也会进入本地静态检查。

**3. 候选 Runtime 生成模块**
候选池只内置一个 `safe_baseline`：完整重算并优先保证正确性，作为所有自动优化失败时的可靠回退。

KV cache、batched decode、预分配 cache、融合投影、SDPA、Triton 等高级方案不再作为固定候选，由 agent 根据当前最佳版本、正确性结果和 benchmark feedback 调用 LLM 动态生成。每个动态候选仍需通过本地预检、正确性测试和 benchmark 后才能参与最终选择。

高分策略是：保留可靠的正确版本，并让 agent 根据实际评测结果持续提出和验证更高级的优化。

**4. LLM Prompt 编排模块**
每轮给 LLM 的不是一句“请优化”，而是结构化上下文：

- 当前模型配置摘要
- 当前 trace/benchmark 摘要
- 当前候选代码的关键片段或 diff
- 正确性失败 case，如果有
- throughput、decode throughput、peak memory
- 明确目标：先修正确性，再优化瓶颈
- 明确限制：保持接口、动态读 config、不能破坏 request state

LLM 输出也要约束成固定格式：

```text
strategy:
expected_benefit:
risk:
patch_or_full_engine:
self_check_notes:
```

agent 解析后再落盘，不直接盲信。

**5. 候选版本管理与快速预检模块**



先把每轮生成的候选保存成 engine_iter_N.py，不要直接覆盖当前 best。
做最低成本检查：能否 import、有没有 create_engine、语法是否正确。

**6. 正确性测试模块**
这是整个系统的第一裁判。顺序建议：

1. 官方 `evaluator/test_correctness.py`
2. agent 自己补充 stress cases：
   - prefill 覆盖旧 request
   - request_ids 非连续、乱序
   - remove 后继续 decode 其他请求
   - 不同 prompt length
   - mixed insert/remove
3. 对失败 case 记录最大误差、失败操作、request id、shape

你直接修改evaluator/test_correctness.py也可以，就走一次测试就行。只要正确性失败，该候选性能记为不可用，LLM 下一轮只收到“修正确性”任务。

**7. Benchmark 与 Profiling 模块**
正确后再测性能：

- 跑官方 `benchmark_throughput.py`
- 分开记录 prefill、decode、mixed
- 记录 tokens/s、decode tokens/s、peak memory
- 如果可用，用 `torch.cuda.Event` 或 `torch.profiler` 粗略定位瓶颈
- trace 可见时，额外计算“真实主要瓶颈”：长 prefill 还是 repeated decode

这个模块产出的数据，是下一轮 LLM 优化 prompt 的核心材料。

**8. 候选选择与回退模块**
维护一个排行榜：

- 正确性通过优先
- 总 tokens/s 最高
- mixed 优先级高于单项极端优化
- 显存不能爆


如果新候选失败，保留旧 best。最终永远把 best candidate 复制到 `workspace/engine.py`，不要让最后一轮失败覆盖可用版本。


**9. 迭代停止条件**
建议设一个时间预算和轮数预算：

- 连续 2-3 轮没有性能提升就停止
- 剩余时间不足时停止冒险优化，固化 best

**10. Output3 报告生成模块**
该模块负责在 agent 结束前生成 workspace/output3.md，作为本次自动化 runtime 生成过程的说明文档。该文档用于展示 agent 如何分析模型、生成候选、执行正确性测试、做性能比较并选择最终 engine.py。需要展示agent的全部运行过程.
输入来源:
环境与输入探测模块：模型配置、权重摘要、设备信息。
Trace 分析模块：观察到的 batch size、prompt length、decode steps、request order。
候选版本管理模块：每轮候选 runtime 的版本号、优化策略、是否通过测试。
正确性测试模块：官方 evaluator/test_correctness.py 的结果。
Benchmark 模块：prefill、decode、mixed 的 tokens/s 和显存占用。
LLM 交互模块：每轮 prompt 的目标摘要、LLM 给出的优化思路、agent 采纳或拒绝的原因。
